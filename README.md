# Расчет ресурсов для кластера Kubernetes

## 1. Расчет ресурсов компонентов приложения
Для начала определим суммарные ресурсы, необходимые для работы всех экземпляров (реплик) приложения.

| Компонент | Кол-во | RAM (на копию) | CPU (на копию) | RAM (итого) | CPU (итого) |
| :--- | :---: | :---: | :---: | :---: | :---: |
| База данных | 3 | 4 ГБ | 1 core | 12 ГБ | 3 cores |
| Система кеширования | 3 | 4 ГБ | 1 core | 12 ГБ | 3 cores |
| Фронтенд | 5 | 50 МБ | 0.2 core | 0.25 ГБ | 1 core |
| Бекенд | 10 | 600 МБ | 1 core | 6 ГБ | 10 cores |
| **Всего (net)** | | | | **30.25 ГБ** | **17 cores** |

## 2. Учет отказоустойчивости (N+1)
Для обеспечения отказоустойчивости при выходе из строя одной ноды, нам необходимо иметь запас ресурсов. При использовании 3-х нод, каждая из них должна нести не более 66% нагрузки, чтобы в случае аварии оставшиеся две смогли принять на себя 100% нагрузки.

*   **Требуемый ресурс CPU с учетом N+1:** 17 cores / 0.66 ≈ **26 cores**
*   **Требуемый ресурс RAM с учетом N+1:** 30.25 ГБ / 0.66 ≈ **46 ГБ**

## 3. Учет системных ресурсов (Kubernetes Overhead)
На каждой ноде необходимо зарезервировать ресурсы для нужд ОС и компонентов Kubernetes (kubelet, kube-proxy, runtime, eviction threshold). Примем стандартные требования для небольшой ноды: **1 core** и **2 ГБ RAM** на каждую ноду.

Для 3 нод:
*   Дополнительно CPU: 3 ноды * 1 core = **3 cores**
*   Дополнительно RAM: 3 ноды * 2 ГБ = **6 ГБ**

## 4. Итоговые требования к кластеру
Суммируем нагрузку, резерв на отказ и системные нужды:

*   **Всего CPU:** 26 (нагрузка+резерв) + 3 (система) = **29 cores**
*   **Всего RAM:** 46 ГБ (нагрузка+резерв) + 6 ГБ (система) = **52 ГБ**

### Итоговая конфигурация нод:
Чтобы обеспечить отказоустойчивость, рекомендую использовать **3 рабочие ноды** со следующими минимальными характеристиками каждой:

*   **Количество нод:** 3
*   **vCPU на ноду:** 10 vCPU (30 всего)
*   **RAM на ноду:** 18 ГБ RAM (54 ГБ всего)

## 5. Рекомендации по деплою
1.  **Anti-Affinity:** Для обеспечения отказоустойчивости БД и Кеша необходимо использовать `podAntiAffinity` (параметр `requiredDuringSchedulingIgnoredDuringExecution`), чтобы реплики одного приложения не оказались на одной ноде.
2.  **Resource Quotas:** В чарте приложения обязательно прописать `resources.requests` и `resources.limits`, чтобы Kubernetes корректно распределял поды по нодам.
3.  **Horizontal Pod Autoscaler:** Для фронтенда и бекенда рекомендуется настроить HPA, так как указанные значения — это базовые требования, а не пиковые.
